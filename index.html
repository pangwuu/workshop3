01

Tensors

A tensor is PyTorch's fundamental data container. Think of it as a typed, n-dimensional array that knows which device it lives on (CPU or GPU) and keeps a record of every operation performed on it — so gradients can flow backwards.

📐

Shape

The dimensions of the tensor. A 2×3 tensor has shape (2, 3). Shape determines how operations compose — matrix multiply, broadcasting, attention — it's all shape arithmetic.

🔢

dtype

The numeric type of every element: float32 for model weights, int64 for token IDs. Mismatched dtypes are the most common runtime error.

⚡

device

Where the data lives: cpu or cuda. Operations between tensors on different devices fail — you must explicitly move tensors with .to(device).

🔗

requires_grad

When True, PyTorch records every operation on this tensor in a computation graph so it can compute derivatives automatically during backpropagation.

Creating Tensors

python
# From data — dtype inferred automatically
data = [[1, 2], [3, 4]]
x_data = torch.tensor(data)           # int64

float_data = [[1.0, 2.0], [3.0, 4.0]]
xf_data = torch.tensor(float_data)    # float32

# From NumPy — SHARED memory, not a copy
np_array = np.array(data)
x_np = torch.from_numpy(np_array)

# Same shape as existing tensor, new values
x_ones = torch.ones_like(x_data)
x_rand = torch.rand_like(x_data, dtype=torch.float)

# From scratch with explicit shape
shape = (2, 3)
rand_tensor  = torch.rand(shape)
ones_tensor  = torch.ones(shape)
zeros_tensor = torch.zeros(shape)
Why shared memory matters torch.from_numpy() and NumPy's t.numpy() do not copy data — they share the same underlying memory buffer. This is fast, but means modifying one modifies the other. Slicing also gives you a view, not a copy. Use .clone() when you need independence.

Key Operations

python
tensor = torch.rand(4, 4)

# Indexing & slicing (returns views)
tensor[0]          # first row
tensor[:, 0]       # first column
tensor[..., -1]    # last column

# cat vs stack — both join, but stack creates a new axis
torch.cat([t1, t2], dim=0)    # (2,3)+(2,3) → (4,3)
torch.stack([t1, t2], dim=0)  # (2,3)+(2,3) → (2,2,3)

# Matrix multiply vs element-wise
y = tensor @ tensor.T          # matmul: (4,4)@(4,4) → (4,4)
z = tensor * tensor            # element-wise: same shape

# Aggregation → scalar tensor → Python float
agg = tensor.sum()
val = agg.item()              # extracts the Python float

Datasets & DataLoaders

PyTorch separates what data is from how to load it. A Dataset describes one sample at a time. A DataLoader wraps it and handles batching, shuffling, and parallel loading — concerns that don't belong in your model code.

🗂

Dataset

Implements __len__ and __getitem__. That protocol is all PyTorch needs — it doesn't care what's inside. This lets you swap datasets without touching training code.

📦

DataLoader

Handles minibatching, shuffling each epoch (reducing overfitting by preventing the model from memorising order), and optional parallel workers via num_workers.

🔄

collate_fn

Called once per batch. Converts a list of individual samples into tensors. Essential for variable-length sequences — like text — where you need to pack or pad before feeding the model.

Custom Dataset Pattern

python
class CustomTextDataset(Dataset):
    def __init__(self, json_file, transform=None, target_transform=None):
        self.labels, self.inputs = [], []
        for line in open(json_file):
            data = json.loads(line.strip())
            for msg, tlabel in zip(data['messages'], data['sender_labels']):
                self.inputs.append(msg)
                # Convert bool → int label (True sender → 0, False → 1)
                self.labels.append(0 if tlabel else 1)
        self.transform = transform
        self.target_transform = target_transform

    def __len__(self):
        return len(self.labels)

    def __getitem__(self, idx):
        text, label = self.inputs[idx], self.labels[idx]
        if self.transform:        text  = self.transform(text)
        if self.target_transform: label = self.target_transform(label)
        return text, label

Text → Tensors: The Vocabulary

Neural networks can't process strings. The vocabulary maps each unique token to an integer ID. The collate_fn then converts a batch of text strings into flat token-ID tensors plus an offsets array that marks where each sequence starts.

python
class Vocab:
    def __init__(self, token_iter):
        self.str_to_num = {"<unk>": 0}  # unknown token always at index 0
        for token in token_iter:
            if token not in self.str_to_num:
                self.str_to_num[token] = len(self.str_to_num)

def collate_batch(batch):
    label_list, text_list, offsets = [], [], [0]
    for _text, _label in batch:
        label_list.append(int(_label))
        tokens = torch.tensor(text_pipeline(_text), dtype=torch.int64)
        text_list.append(tokens)
        offsets.append(tokens.size(0))  # length of this sequence
    # cumsum converts lengths [0,5,3,7] → start indices [0,5,8,15]
    offsets = torch.tensor(offsets[:-1]).cumsum(dim=0)
    return torch.cat(text_list), torch.tensor(label_list), offsets
Why flatten + offsets instead of padding? Padding packs all sequences to the same length with dummy tokens, which wastes compute. nn.EmbeddingBag uses the offsets trick instead — it concatenates all sequences into one flat tensor and uses the offsets array to know where each begins. No wasted computation, no padding embeddings polluting your mean.
03

Building the Neural Network

Every PyTorch model is a subclass of nn.Module. You define the layers in __init__ and the forward pass — how data flows through those layers — in forward(). PyTorch's autograd then automatically handles the backward pass.

Architecture

Input
Token IDs [flat int64] + offsets
Layer 1
EmbeddingBag vocab → 64 mode=mean
Layer 2
Linear 64 → num_class Wx + b
Output
Logits [batch, 2] raw scores
python
class TextClassificationModel(nn.Module):
    def __init__(self, vocab_size, embed_dim, num_class):
        super().__init__()
        self.embedding = nn.EmbeddingBag(vocab_size, embed_dim, sparse=False)
        self.fc = nn.Linear(embed_dim, num_class)
        self.init_weights()

    def init_weights(self):
        initrange = 0.5
        self.embedding.weight.data.uniform_(-initrange, initrange)
        self.fc.weight.data.uniform_(-initrange, initrange)
        self.fc.bias.data.zero_()

    def forward(self, text, offsets):
        embedded = self.embedding(text, offsets)  # → [batch, embed_dim]
        return self.fc(embedded)                  # → [batch, num_class]
🧩

EmbeddingBag

Maps each token ID to a learned vector (embedding), then averages all vectors in the sequence. The average becomes the document representation. No word order — a bag-of-words assumption, but learned.

➗

Linear (fc)

A weight matrix W and bias b. Computes xWT + b. The learned weights in W determine which embedding dimensions matter for each class.

📊

Logits

Raw, unnormalised scores. Not probabilities yet. CrossEntropyLoss converts them internally via softmax. You only apply softmax explicitly at inference time when you need probabilities.

🏁

init_weights

Uniform initialisation in [−0.5, 0.5] breaks symmetry. If all weights started at zero, every neuron would compute identical gradients and learn identically — the network would never specialise.

Task 1 — Add a Hidden Layer

Task 1: Hidden Layer with Tanh Activation

  1. Use an EmbeddingBag to represent words
  2. Use a Linear layer to project to 100 hidden dimensions
  3. Apply torch.tanh as the activation function
  4. Use a second Linear layer to get scores across labels
Why an activation function? Without tanh (or any non-linearity), stacking two linear layers is mathematically identical to one linear layer — the composition of linear functions is linear. The activation function introduces non-linearity, letting the hidden layer learn complex feature interactions that a single matrix can't represent.
python — Task 1 solution
class TextClassificationModelWithHidden(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim, num_class):
        super().__init__()
        self.embedding = nn.EmbeddingBag(vocab_size, embed_dim, sparse=False)
        self.fc1 = nn.Linear(embed_dim, hidden_dim)   # 64 → 100
        self.fc2 = nn.Linear(hidden_dim, num_class)   # 100 → 2
        self.init_weights()

    def init_weights(self):
        initrange = 0.5
        for layer in [self.embedding, self.fc1, self.fc2]:
            layer.weight.data.uniform_(-initrange, initrange)
        self.fc1.bias.data.zero_()
        self.fc2.bias.data.zero_()

    def forward(self, text, offsets):
        embedded = self.embedding(text, offsets)  # [batch, 64]
        hidden   = torch.tanh(self.fc1(embedded)) # [batch, 100]
        return self.fc2(hidden)                    # [batch, 2]

model_hidden = TextClassificationModelWithHidden(
    vocab_size, emsize, hidden_dim=100, num_class=num_class
).to(device)
04

Autograd & Backpropagation

When you set requires_grad=True on a tensor, PyTorch silently builds a directed acyclic graph (DAG) as operations run. Each node stores how to compute its derivative. When you call .backward(), the engine walks this graph in reverse, applying the chain rule at each step to accumulate gradients in each parameter's .grad attribute.

Computation Graph
python
x = torch.ones(5)
y = torch.zeros(3)
w = torch.randn(5, 3, requires_grad=True)
b = torch.randn(3,    requires_grad=True)

z    = x @ w + b
loss = nn.functional.binary_cross_entropy_with_logits(z, y)

loss.backward()   # walks the DAG; fills .grad on w and b
print(w.grad)       # ∂loss/∂w — shape (5,3)
print(b.grad)       # ∂loss/∂b — shape (3,)

# Disable gradient tracking for inference
with torch.no_grad():
    z = x @ w + b    # no DAG built — faster, less memory
Why zero_grad() before each batch? Gradients accumulate in .grad by default — each .backward() call adds to the existing value. This is occasionally useful (gradient checkpointing) but almost always a bug in training loops. Call optimizer.zero_grad() at the start of every batch to clear them first.
05

Training the Model

Training is an iterative loop: predict → measure error → compute gradients → update weights. Each pass over the full dataset is an epoch. Each weight update uses a minibatch — a tradeoff between the accuracy of full-batch gradients and the speed of single-sample updates.

STEP 01
Forward Pass

Feed a batch through the model. Get predicted logits. The computation graph is built silently in the background.

STEP 02
Compute Loss

CrossEntropyLoss softmaxes the logits and computes negative log-likelihood against the true labels. Lower is better.

STEP 03
Backward Pass

loss.backward() walks the DAG backwards. Each parameter's .grad is filled with how much it contributed to the loss.

STEP 04
Clip Gradients

clip_grad_norm_(0.1) caps gradient magnitude. Prevents exploding gradients from destabilising training — a common issue in RNNs and early in training.

STEP 05
Optimizer Step

SGD nudges each weight in the direction that decreases loss: w ← w − lr × ∂loss/∂w. The learning rate controls step size.

STEP 06
Zero Gradients

optimizer.zero_grad() clears the .grad buffers so the next batch starts fresh. This is called at the start of the next iteration.

python — train loop
def train_loop(dataloader, epoch, loss_fn, optimizer, model):
    model.train()                            # enables dropout/batchnorm training behaviour
    total_acc, total_count = 0, 0

    for idx, (text, label, offsets) in enumerate(dataloader):
        optimizer.zero_grad()               # clear last batch's gradients
        predicted_label = model(text, offsets)
        loss = loss_fn(predicted_label, label)
        loss.backward()                      # compute gradients
        torch.nn.utils.clip_grad_norm_(model.parameters(), 0.1)
        optimizer.step()                     # apply gradients

        total_acc += (predicted_label.argmax(1) == label).sum().item()
        total_count += label.size(0)

def test_loop(dataloader, loss_fn, model):
    model.eval()                             # disables dropout etc.
    total_acc, total_count = 0, 0
    with torch.no_grad():                    # no DAG needed — saves memory
        for text, label, offsets in dataloader:
            predicted_label = model(text, offsets)
            total_acc += (predicted_label.argmax(1) == label).sum().item()
            total_count += label.size(0)
    return total_acc / total_count

Hyperparameters & the Scheduler

📉

Learning Rate = 5

SGD with a high LR converges fast but risks overshooting. The StepLR scheduler multiplies LR by 0.1 when validation accuracy stops improving — annealing it to fine-tune the minimum.

🔀

shuffle=True

Reshuffling each epoch prevents the model from memorising presentation order. If your dataset is sorted by label, training without shuffle would yield misleading loss curves.

📏

batch_size = 64

Larger batches give more accurate gradient estimates but need more memory. Smaller batches introduce noise — which can actually help escape sharp local minima and generalise better.

python — scheduler + epoch loop
loss_fn   = torch.nn.CrossEntropyLoss()
optimizer = torch.optim.SGD(model.parameters(), lr=5)
scheduler = torch.optim.lr_scheduler.StepLR(optimizer, 1.0, gamma=0.1)
total_accu = None

for epoch in range(1, epochs + 1):
    train_loop(train_dl, epoch, loss_fn, optimizer, model)
    accu_val = test_loop(valid_dl, loss_fn, model)
    if total_accu is not None and total_accu > accu_val:
        scheduler.step()    # decay LR if val accuracy dropped
    else:
        total_accu = accu_val
06

Saving & Loading

💾

state_dict only (recommended)

Saves just the weights as a plain dict. To load, you recreate the model architecture in code first, then load weights into it. Portable across Python versions.

📦

Full model (pickle)

Saves architecture + weights via pickle. Convenient but fragile — the class definition must exist at load time, and the file can execute arbitrary code (weights_only=False).

python
# Save weights only (recommended)
torch.save(model.state_dict(), 'model_weights.pth')

new_model = TextClassificationModel(vocab_size, emsize, num_class)
new_model.load_state_dict(torch.load('model_weights.pth'))
new_model.eval()    # IMPORTANT: disables training-only layers

# Save full model (less portable)
torch.save(model, 'model.pth')
model = torch.load('model.pth', weights_only=False)  # ⚠ trust the file
Why call model.eval()? Some layers behave differently at inference time. Dropout randomly zeros activations during training (regularisation) but should be deterministic at inference. BatchNorm uses running stats instead of batch stats. model.eval() switches these modes. Forgetting it causes inconsistent predictions.
